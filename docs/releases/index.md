## リリースページ本文の構成（推奨）

GitHub Releases の本文は、以下の構成で記載します。

1. **新機能**
2. **改善**
3. **バグ修正**
4. **既知の問題**（該当がある場合のみ）
5. **その他**（依存関係更新・運用変更など）

### 書き方の基準

- 1項目=1行で簡潔に書く（ユーザーが一目で理解できる粒度）。
- まず「何ができるようになったか」を書き、その後に補足理由を書き足す。
- 技術用語は必要最小限にし、ユーザー視点の効果を優先する。
- 機能名が紛らわしい場合は、UI上の名称に合わせる。

## 記載漏れ防止チェックリスト

- 追加された主要機能が網羅されているか（新規導線・新規画面・新規連携）。
- 体験が変わる改善点（速度/安定性/表示/導線）を記載しているか。
- 影響が広いバグ修正（クラッシュ/データ破損/表示不具合）を記載しているか。
- 外部連携やOS連携の挙動変更があれば必ず記載しているか。
